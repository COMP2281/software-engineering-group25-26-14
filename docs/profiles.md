## Vehicle Profile Calibration (Feature 3)

The vehicle profile calibration module resides in `data_pipeline/profiles/` and represents "Feature 3" of our data product. 

This module takes a `ProcessedDataset` generated by the Preprocessing Pipeline and produces a `VehicleProfile`, a Digital Twin object. The profile contains comprehensive mechanical and behavioural specs which will be essential for subsequent Highly Accurate Fuel Estimation processing.

### Architecture and Logic

The module separates concern into the following key layers:
1. `models.py`: Defines the `VehicleProfile` and `VehicleSpecs` dataclasses, enforcing strong, predictable structures with total operational weight aggregation built in.
2. `resolver.py` (`VehicleSpecResolver`): Represents the external integration layer. It queries the **Cars API by API Ninjas via RapidAPI** to lookup make/model vehicle data dynamically. 
   - **Data Extracted**: It explicitly parses literal `displacement` logic (engine volume in Liters) and `fuel_type` mapping ("Petrol", "Diesel", "Electric").
   - **Mathematical Inference**: It recalculates bounds like stoichiometric AFR (Air-Fuel Ratio) based on the fuel type.
   - **Fallback Logic**: Because the Cars API does not return every single deep engineering spec (like base weight or redline), and API limits can be hit, this module utilizes a robust **fallback proxy logic** containing sensible European industry defaults (e.g. 14.7 AFR for Petrol, 2.0L Engine, 1400kg base weight).
3. `builder.py` (`VehicleProfileBuilder`): The entry-point API. It handles the data fusion between:
   * **Engine properties** obtained via `VehicleSpecResolver` (or its fallback).
   * **Driver behavior** derived entirely from the aggregated standard pandas DataFrame records from the `ProcessedDataset` (Telematics extraction).

#### Telematics Extraction for Dynamic Calibration
Instead of strictly relying on static API definitions, the Builder enriches the profile's accuracy by calibrating to the actual physical conditions extracted from the telematics data:
- **Idle RPM Baseline (`idle_rpm_baseline`)**: Averaged RPM measured precisely when `Speed == 0` within the processed data. This tells us exactly where this specific engine rests.
- **Typical Maximum RPM (`typical_max_rpm`)**: Calculated at the `99th percentile` for all RPM readings in the driving trips, successfully eliminating noise/spikes and capturing the actual behavioral operational ceiling of the driver.
- **Static vs. Behavior Match (`mechanical_max_rpm`)**: While driver behavior sets the `typical_max_rpm`, the engine constraints set by the API define the absolute `mechanical_max_rpm` preventing fuel-burn algorithms from overextrapolating.
- **Dynamic Load Management (`dynamic_payload_kg`)**: Allows initialization parameters such as `passenger_count` and `cargo_weight_kg` that augment the Digital Twin's dynamic payload. This enables algorithms to penalize or adjust fuel efficiency dynamically based on extra load.

### Future Improvements (Rich API Integration)
This architecture is intentionally modular and lays the groundwork for high-fidelity data replacement. If budgets or requirements allow in the future, utilizing a more precise and rich premium API (or querying deep NHTSA databases with explicit VIN numbers) could vastly enrich the vehicle model. Providing accurate exact values for `curb_weight`, `mechanical_redline`, `exact AFR arrays`, and `transmission ratios` will plug seamlessly into this existing structure without breaking subsequent pipeline analysis.

### API/Programmatic Usage

Integrating the profiler into other scripts is designed to be highly intuitive. 

```python
from data_pipeline.profiles.builder import VehicleProfileBuilder

# Assume `processed_dataset` object was returned by the PreprocessingPipeline
builder = VehicleProfileBuilder(api_timeout=5)

# Calculate dynamic payload (optional kwargs are evaluated here)
profile = builder.build_from_dataset(
    processed_dataset, 
    passenger_count=2,       # assumes 75kg per passenger
    cargo_weight_kg=150.0
)

# The vehicle profile provides easy property access
print(f"Active payload logic calculates total weight at {profile.total_weight_kg}kg")
print(f"Vehicle Fuel Setup: {profile.fuel_type}, AFR is {profile.stoichiometric_afr}")
print(f"Driver usually pushes up to {profile.typical_max_rpm} RPM, while mechanical max is {profile.mechanical_max_rpm}")
```
