## Vehicle Profile Calibration (Feature 3)

The vehicle profile calibration module resides in `data_pipeline/profiles/` and represents "Feature 3" of our data product. 

This module takes a `ProcessedDataset` generated by the Preprocessing Pipeline and produces a `VehicleProfile`, a Digital Twin object. The profile contains comprehensive mechanical and behavioural specs which will be essential for subsequent Highly Accurate Fuel Estimation processing.

### Architecture and Logic

The module separates concern into the following key layers:
1. `models.py`: Defines the `VehicleProfile` and `VehicleSpecs` dataclasses, enforcing strong, predictable structures with total operational weight aggregation built in.
2. `exceptions.py`: Defines custom errors (`CalibrationDataError`, `VehicleResolutionError`) for clear downstream exception handling.
3. `resolver.py` (`VehicleSpecResolver`): Represents the external integration layer. It hits the free NHTSA vPIC API to lookup make/model vehicle models dynamically. Because European/UK models frequently lack deep engine specifications (base weight, fuel type, mechanical redline, etc.) via public make-model endpoints, this module utilizes a robust **fallback proxy logic** containing sensible European industry defaults (e.g. 14.7 AFR for Petrol, 2.0L Engine).
4. `builder.py` (`VehicleProfileBuilder`): The entry-point API. It handles the data fusion between:
   * Engine properties obtained via `VehicleSpecResolver`
   * Driver behavior derived entirely from the aggregated DataFrame records from `ProcessedDataset` (Telematics extraction).

#### Telematics Extraction & Correction
- **Idle RPM Baseline**: Averaged RPM measured precisely when `Speed == 0` within the processed data.
- **Typical Maximum RPM**: Calculated at the `99th percentile` for all RPM readings, successfully eliminating noise and capturing the actual behavioural operational ceiling.
- **Dynamic Load Management**: Allows initialization parameters such as `passenger_count` and `cargo_weight_kg` that augment the Digital Twin's dynamic payload.

### API/Programmatic Usage

Integrating the profiler into your scripts is designed to be highly intuitive. 

```python
from data_pipeline.profiles.builder import VehicleProfileBuilder

# Assume `processed_dataset` object was returned by the PreprocessingPipeline
builder = VehicleProfileBuilder(api_timeout=5)

# Calculate dynamic payload (optional kwargs are evaluated here)
profile = builder.build_from_dataset(
    processed_dataset, 
    passenger_count=2,       # assumes 75kg per passenger
    cargo_weight_kg=150.0
)

# The vehicle profile provides easy property access
print(f"Active payload logic calculates total weight at {profile.total_weight_kg}kg")
print(f"Vehicle Fuel Setup: {profile.fuel_type}, AFR is {profile.stoichiometric_afr}")
print(f"Driver usually pushes up to {profile.typical_max_rpm} RPM, while mechanical max is {profile.mechanical_max_rpm}")
```

### Virtual Environment Setup & Execution

When actively debugging or altering features logic, run the module via the virtual environment. Ensure dependencies (`pandas`, `requests`) are up-to-date.

```bash
# Set up Venv (if not done previously)
python -m venv .venv
source .venv/bin/activate
pip install requests pandas

# Given it has an __name__ == '__main__' execution block in the builder, it serves as a self-contained integration test:
python -m data_pipeline.profiles.builder 
```

The script execution will execute placeholder data representing telematics readouts and mock a missing NHTSA endpoint representation, thereby actively exercising both the DataFrame operations, default resolution logic and print out the synthesized properties.
